# Harvest Coffee Mobile Code + Security Review

Date: 2026-06-26
Scope: `apps/mobile`, shared `packages/api`, shared `packages/domain`, and the web proxy/auth bridge used by mobile.

## Executive Summary

The mobile app is in a much better state than the earlier mock/demo phase: mobile no longer contains a mock runtime fallback, no Base44 API key or client secret was found in the mobile/shared source, and live data goes through the web proxy contract. The main risks are now around OAuth deep-link hardening, token transport, and mutation flows that can report a failed request after Base44 has already created or updated data.

No Critical findings were identified from static review. The High item should be fixed before production mobile auth is trusted outside development/test devices.

## Positive Controls

- Mobile uses `EXPO_PUBLIC_HARVEST_API_URL` only as a public proxy URL; no Base44 secret is shipped in the app.
- `apps/mobile/lib/mobile-state.tsx` creates the API client through `createProxyHarvestApi` and fails closed when the proxy URL is missing.
- No `Alert.alert`, `WebView`, `eval`, `dangerouslySetInnerHTML`, `localStorage`, `AsyncStorage`, `SecureStore`, `api_key`, or `client_secret` usage was found in `apps/mobile`, `packages/api`, or `packages/domain`.
- Root routing is centralized in `apps/mobile/app/_layout.tsx`; logged-out private routes redirect to login and role shells are selected from one place.
- Destructive product, rental, address, and notification deletes use `ConfirmDialog` and loading/disabled state in the inspected screens.
- Admin/dealer data access is primarily enforced by the web proxy route, including `created_by_id` order access checks and admin write guards.

## Findings

### High

#### H-01 Mobile OAuth callback forwards access tokens to broadly allowed Expo deep links

Evidence:
- `apps/web/app/mobile-auth/callback/route.ts:1` allows `exp:`, `exps:`, and `harvestcoffee:` protocols.
- `apps/web/app/mobile-auth/callback/route.ts:3` through `apps/web/app/mobile-auth/callback/route.ts:22` copies `access_token` into the redirect URL query.
- `apps/web/app/mobile-auth/callback/route.ts:25` through `apps/web/app/mobile-auth/callback/route.ts:35` validates only the URL protocol.
- `apps/mobile/app/(public)/login.tsx:50` through `apps/mobile/app/(public)/login.tsx:55` builds the mobile auth bridge with a `return_to` URL.

Impact:
If an attacker can initiate a login flow with a crafted `return_to`, the callback can forward a bearer token to an arbitrary `exp:` or `exps:` target. This is useful during Expo development, but it is too broad for production auth.

Recommended fix:
- Use the custom app scheme `harvestcoffee:` for production mobile callbacks.
- Keep `exp:`/`exps:` only behind an explicit development flag or exact allowlist.
- Add a nonce/state generated by the mobile app and validated after callback.
- Prefer a short-lived code exchange over putting the final access token in the redirect URL.

Test scenario:
- Start a login with `return_to=exp://attacker.example/--/login` and verify the bridge rejects it.
- Start a valid login with the expected Expo development URL and a matching nonce and verify it succeeds.

### Medium

#### M-01 Proxy bearer token is sent in the JSON request body

Evidence:
- `packages/api/src/index.ts:391` through `packages/api/src/index.ts:440` posts `{ action, input, token }` to the proxy.
- `apps/web/app/api/harvest/route.ts:60` through `apps/web/app/api/harvest/route.ts:81` reads `token` from the request body.

Impact:
Bearer tokens in JSON bodies are easier to expose through body logging, error captures, and proxy debugging. Headers are not magic, but infrastructure commonly redacts `Authorization` more reliably than arbitrary JSON fields.

Recommended fix:
- Send `Authorization: Bearer <token>` from `createProxyHarvestApi`.
- Have `/api/harvest` read the header first, with body token support removed after migration.
- Ensure request bodies are never logged with token fields.

Test scenario:
- Create an order from mobile and inspect the proxy request. The body must contain only `action` and `input`; the token must be in the `Authorization` header.

#### M-02 Successful writes are coupled to full refresh calls

Evidence:
- `apps/mobile/lib/mobile-state.tsx:217` through `apps/mobile/lib/mobile-state.tsx:221` creates an order and then awaits `refreshDealerData`.
- `apps/mobile/lib/mobile-state.tsx:223` through `apps/mobile/lib/mobile-state.tsx:228` creates a rental and then awaits `refreshDealerData`.
- `apps/mobile/app/(public)/products.tsx:65` through `apps/mobile/app/(public)/products.tsx:93` treats any thrown error as order failure.
- `apps/mobile/app/create-rental.tsx:84` through `apps/mobile/app/create-rental.tsx:110` has the same create-then-refresh pattern.

Impact:
Base44 can successfully create an order or rental, but a follow-up refresh failure makes the UI show a failed request. That can leave the cart/form in place, encourage duplicate submissions, or trigger confusing navigation behavior.

Recommended fix:
- Return success immediately after the create call succeeds.
- Refresh in the background or in a separate best-effort step.
- If refresh fails, show a non-blocking warning like "Created, but the latest list could not be refreshed."

Test scenario:
- Mock or force `refreshDealerData` to fail after a successful `createOrder`. The app should still clear the cart and navigate to order detail.

#### M-03 Admin settings are session-only but presented as an admin settings screen

Evidence:
- `apps/mobile/app/admin-settings.tsx:14` through `apps/mobile/app/admin-settings.tsx:18` says preferences remain in the admin session.
- `apps/mobile/lib/mobile-state.tsx:274` through `apps/mobile/lib/mobile-state.tsx:280` updates only local `currentUser` and `users`.
- User API docs did not show an `admin_settings` field as a stable Base44 user field.

Impact:
Admins may believe notification/reminder settings persist in Base44, but the values are lost on app restart and are not guaranteed to affect backend behavior.

Recommended fix:
- Keep this screen explicitly marked as local/session-only, or disable live-looking controls until a supported Base44 entity/function exists.
- If persistence is required, add a documented settings entity or Base44 function and wire it through the proxy.

Test scenario:
- Change an admin setting, restart the app, and verify either the setting is persisted through a supported API or the UI clearly says it is not.

#### M-04 Product "Generate description" is local, not a Base44 AI/function call

Evidence:
- `apps/mobile/app/admin-products.tsx:141` through `apps/mobile/app/admin-products.tsx:158` calls `draftProductDescription(current)` and displays "Description drafted locally."

Impact:
The UI may look like a live AI generation feature, but it is a local template. This is an API parity gap if the web/Base44 product has a real "generate with AI" behavior.

Recommended fix:
- Add a proxy action for the documented Base44 generation function if available.
- Otherwise keep the label explicit, for example "Draft locally", so users do not expect a live AI request.

Test scenario:
- Tap the generate button and confirm whether a network function call is expected. If expected, verify it happens through the proxy.

#### M-05 Public products can fail noisily when the proxy URL is missing

Evidence:
- `apps/mobile/app/(public)/products.tsx:43` through `apps/mobile/app/(public)/products.tsx:59` loads public products when logged out.
- `apps/mobile/lib/mobile-state.tsx:428` through `apps/mobile/lib/mobile-state.tsx:435` rejects API calls when `EXPO_PUBLIC_HARVEST_API_URL` is missing.

Impact:
Opening public products without a configured proxy can create an unhandled promise rejection instead of a clean configuration error state.

Recommended fix:
- Add `.catch` around the public product load and show a native error state.
- Optionally skip public product fetching entirely when no proxy endpoint is configured.

Test scenario:
- Start the app without `EXPO_PUBLIC_HARVEST_API_URL`, open public products, and verify the app shows a controlled error instead of console/runtime noise.

### Low

#### L-01 Access token is memory-only with no explicit expiry handling

Evidence:
- `apps/mobile/lib/mobile-state.tsx:19` stores `mobileAccessToken` in module memory.
- `apps/mobile/lib/mobile-state.tsx:182` through `apps/mobile/lib/mobile-state.tsx:200` sets the token after login.
- `apps/mobile/lib/mobile-state.tsx:202` through `apps/mobile/lib/mobile-state.tsx:215` clears it on logout.

Impact:
Memory-only storage is safer than persistent storage for theft resistance, but users lose session on app restart and stale/expired token behavior depends on later API failures.

Recommended fix:
- Decide the product policy: memory-only for safest auth, or `SecureStore` with expiry and 401 handling for persistent login.
- If persistence is added, store only the minimal token, clear it on logout, and clear/re-auth on 401.

Test scenario:
- Login, kill/reopen the app, and verify the resulting behavior matches the chosen policy.

#### L-02 Order detail can show stale cached order data

Evidence:
- `apps/mobile/app/order/[id].tsx:15` through `apps/mobile/app/order/[id].tsx:34` only fetches the order if it is not already present in local state.

Impact:
If an admin changes status/payment/tracking on another device, mobile order detail can show the cached version until another refresh happens.

Recommended fix:
- Show cached data immediately, then revalidate order detail on screen focus.
- Keep a loading or "updated" state separate from the cached view.

Test scenario:
- Open an order detail, update the same order elsewhere, navigate away/back, and verify fresh status is shown.

#### L-03 Notification mark-read failures are silent

Evidence:
- `apps/mobile/app/notifications.tsx:14` through `apps/mobile/app/notifications.tsx:21` catches no error in `markRead`.

Impact:
If a mark-read request fails, the user gets no message and the badge/list can look inconsistent.

Recommended fix:
- Catch errors and show `StatusBanner`, matching delete notification behavior.

Test scenario:
- Force `markNotificationRead` to return 403/500 and verify a visible error appears.

#### L-04 Unknown Base44 roles map to dealer

Evidence:
- `packages/api/src/index.ts:750` through `packages/api/src/index.ts:753` maps `admin` to admin and all other roles to dealer.

Impact:
This matches the known Base44 `user` role to the app's dealer role, but it fails open for unexpected role values.

Recommended fix:
- Map `user` explicitly to dealer.
- Treat unknown roles as unauthorized or a limited default until reviewed.

Test scenario:
- Simulate a user with role `guest` or an empty role and verify private dealer screens are not granted automatically.

## Screen Flow Notes

Dealer:
- Products/cart uses profile addresses and disables the place-order button while saving.
- Orders/order detail/track order are routed through the role-aware shell.
- Rentals/create rental use native date picker and proxy create flows.
- Profile address add/edit/delete/select uses `updateCurrentUser` and delete confirmation.
- Notifications can mark read/delete, but mark-read needs visible error handling.

Admin:
- Dashboard/orders/products/stock/customers/reports/settings/rentals/calendar use the shared state/proxy model.
- Customer order summaries use `createdById` first through `getOrdersForUser`.
- Product/rental/address destructive actions have confirmation.
- Admin settings and local product description generation are the main API parity gaps.

## Base44 Project Action Required

- Notification side effects should stay in the Base44 project/functions. Mobile should read, mark read, and delete notifications, but should not create duplicate notification records to compensate for Base44 side-effect bugs.
- Order-created and order-status-changed notification correctness depends on the Base44 side effect assigning the right recipient fields.
- Rental status automation and reminder notification behavior should be validated in Base44 functions/schedules, not faked in mobile.

## Recommended Fix Order

1. Harden mobile OAuth callback: exact production scheme, state/nonce, and development-only `exp:` allowlist.
2. Move proxy token transport from JSON body to `Authorization` header.
3. Decouple successful writes from post-write refresh failures.
4. Decide and implement token persistence/expiry policy.
5. Resolve admin settings and product description generation parity.
6. Add stale detail revalidation and notification mark-read error messaging.
